1st prompt was:

You are working in a blank Git repository named OpenMES.

Project goal:
Build OpenMES, an open-source .NET-based shop-floor execution portal for manufacturing operators, technicians, quality staff, and supervisors. This is not intended to replace an ERP system. It should sit in front of ERP/MRP/database/document systems and provide a clean operator-level interface for:

- Viewing assigned and upcoming jobs
- Opening correct work instructions, setup sheets, drawings, and quality documents
- Scanning labels and issuing material lots to jobs
- Recording production quantities, scrap, downtime, and quality checks
- Showing schedules by machine/resource/work center
- Providing extensibility points for company-specific ERP/database/document/barcode integrations

Primary design principle:
OpenMES should hide ERP complexity from operators while remaining easy for internal company developers to extend.

Target stack:
- C# / .NET 10 if available
- ASP.NET Core
- Blazor Web App for the UI
- Entity Framework Core
- PostgreSQL as the primary database
- Docker Compose for local development
- SignalR for future realtime updates
- xUnit or NUnit for tests

If .NET 10 is not available in the environment, use the newest installed LTS SDK and document the deviation in TASKS.md and CHANGELOG.md.

Do not try to build a complete ERP or a full heavyweight MES immediately. Start with a clean vertical slice:
1. Jobs
2. Parts and revisions
3. Resources/work centers
4. Documents linked to part/revision/operation
5. Operator job page
6. Material scan/issue workflow
7. Production event logging

Architecture rules:
- Keep domain entities in OpenMES.Domain.
- Keep use cases/application services in OpenMES.Application.
- Keep EF Core, database, filesystem, and external integrations in OpenMES.Infrastructure.
- Keep Blazor UI in OpenMES.Web.
- Keep background sync/export logic in OpenMES.Worker.
- Keep plugin interfaces in OpenMES.PluginAbstractions.
- Do not let the UI depend directly on ERP schemas.
- Normalize external data into OpenMES concepts.
- Treat production activity as event/audit records, not just mutable status fields.

Document handling:
For the MVP, support documents as records with file paths or URLs. Do not build a full document management system yet.

A document should have:

- Title
- DocumentType
- PartNumber optional
- Revision optional
- OperationCode optional
- ResourceCode optional
- EffectiveDate optional

- IsObsolete

Document resolver behavior:
Given a job, part revision, operation, and resource, resolve the relevant released documents and show them on the job page.

Production tracking:
Production events should be append-only records. For now, store:
- Event id
- Event type
- Job id
- Resource id optional
- Operation id optional
- User/operator placeholder
- Quantity optional
- Reason code optional
- Notes optional
- Timestamp UTC
- Raw scan value optional

Do not overwrite production history.

Material issue workflow:
Implement a simple workflow:
1. Open job
2. Go to scan page
3. Enter or scan material lot barcode
4. Parse barcode
5. Validate that it looks like a material/lot scan
6. Create MaterialIssued production event
7. Show confirmation
